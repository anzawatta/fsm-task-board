# EARS-008: MCP Write Tools

**Status:** Draft  
**Date:** 2026-05-27

MCP サーバー（`canvas_reader_mcp.py`）を通じたキャンバスへの書き込み操作に関する要件。
アクターは **MCP tools**（ブラウザ UI は EARS-001〜EARS-002 が担当）。

---

## ツール一覧

| Tool | Signature | 説明 |
|------|-----------|------|
| `add_node` | `(filename, name, status?, dod?)` | ノードを追加し、採番・自動配置を行う |
| `update_node` | `(filename, id, name?, status?)` | 既存ノードの name または status を更新する |
| `add_edge` | `(filename, from_node, to_node, label?)` | エッジを追加する |
| `change_edge` | `(filename, edge_id, from_node?, to_node?, label?)` | エッジを in-place で書き換える（ID は保持） |
| `remove_node` | `(filename, id)` | ノードとその接続エッジをカスケード削除する |
| `remove_edge` | `(filename, id)` | エッジを削除する |

> `filename` は `_safe_path()` を通じて検証される。座標・サイズ引数は受け付けない（自動配置はツール責任）。

---

## 不変条件

1. REQ-U001: Each write tool SHALL re-read the canvas file before computing new IDs (no cached counters across tool calls)
2. REQ-U002: Node IDs generated by `add_node` SHALL follow the pattern `s{N}` where N = max numeric suffix of existing node IDs + 1
3. REQ-U003: Edge IDs generated by `add_edge` SHALL follow the pattern `e{N}` where N = max numeric suffix of existing edge IDs + 1
4. REQ-U004: Node `status` passed to `add_node` or `update_node` SHALL be one of: `null` (未着手), `"wip"` (実装中), `"done"` (完了), consistent with EARS-003 REQ-U001
5. REQ-U005: The `fromNode` and `toNode` of any edge SHALL reference node IDs that exist in the canvas at the time of the write
6. REQ-U006: Node `name` SHALL NOT exceed 80 characters; edge `label` SHALL NOT exceed 80 characters
7. REQ-U007: All write operations SHALL be atomic — the tool SHALL write to a temporary file and rename it to the target path
8. REQ-U008: Before each write, the tool SHALL save a backup of the pre-write canvas to `SNAPSHOT_DIR/writes/<filename>.<timestamp>.json`
9. REQ-U009: The write backup path (`SNAPSHOT_DIR/writes/`) SHALL be distinct from the read-diff snapshot path used by `read_canvas` — the two paths SHALL NOT overlap
10. REQ-U010: Before each write, if the canvas file's mtime has changed since the MCP server last read it, the write SHALL be refused unless `force=True` is passed by the caller
11. REQ-U011: All `filename` arguments SHALL be validated through `_safe_path()` before any file access

---

## 敵対条件

1. REQ-W001: System SHALL NOT write `status` values other than `null`, `"wip"`, or `"done"` — any other value SHALL result in `{"status":"error","reason":"invalid status","conflicting_id":null}` being returned (consistent with EARS-003 REQ-W001)
2. REQ-W002: System SHALL NOT create an edge whose `fromNode` does not exist in the canvas at write time — SHALL return `{"status":"error","reason":"fromNode not found","conflicting_id":"<id>"}`
3. REQ-W003: System SHALL NOT create an edge whose `toNode` does not exist in the canvas at write time — SHALL return `{"status":"error","reason":"toNode not found","conflicting_id":"<id>"}`
4. REQ-W004: System SHALL NOT generate a node ID that already exists in the canvas (consistent with EARS-001 REQ-W002)
5. REQ-W005: System SHALL NOT generate an edge ID that already exists in the canvas (consistent with EARS-001 REQ-W003)
6. REQ-W006: System SHALL NOT write to the canvas file non-atomically — direct write without temp-then-rename is prohibited
7. REQ-W007: System SHALL NOT overwrite a canvas file that has been modified since the MCP last read it, unless `force=True` is explicitly passed — SHALL return `{"status":"error","reason":"file modified since last read"}`
8. REQ-W008: System SHALL NOT accept a node `name` or edge `label` exceeding 80 characters — SHALL return `{"status":"error","reason":"name exceeds 80 characters"}` or `{"status":"error","reason":"label exceeds 80 characters"}`

---

## State-driven requirements

1. REQ-S001: While a node with `status: "wip"` exists in the canvas, `add_node` and `update_node` SHALL preserve that status unchanged unless the caller explicitly provides a new value
2. REQ-S002: While `force=True` is passed, the mtime check SHALL be bypassed and the write SHALL proceed regardless of concurrent modification

---

## Event-driven requirements

1. REQ-E001: When `add_node` is called, the system SHALL compute `x = max(existing node x + width) + 20` and `y = median(existing node y values)` (or `0` if no nodes exist), assign them to the new node, and return the full created node object with `"status":"created"`
2. REQ-E002: When `add_edge` is called with valid `from_node` and `to_node`, the system SHALL create the edge, save atomically, write a backup, and return the full created edge object with `"status":"created"`
3. REQ-E003: When `change_edge` is called, the system SHALL update `fromNode`, `toNode`, and/or `label` in-place on the existing edge (keeping the edge ID), then save atomically
4. REQ-E004: When `remove_node` is called, the system SHALL delete the target node and also delete all edges whose `fromNode` or `toNode` matches the removed node ID (cascade delete)
5. REQ-E005: When `remove_edge` is called, the system SHALL delete only the specified edge, leaving connected nodes intact
6. REQ-E006: When `update_node` is called with a `name` or `status` argument, the system SHALL update only the supplied fields and leave all other node fields (including `dod`) unchanged

---

## Advisory [ADV]

- **ADV-001** (auto-placement layout degradation): `add_node` の自動配置は `x = max(existing x+width)+20` で計算するため、ノードを追加し続けると水平方向に伸びていく。ユーザーはブラウザ UI でノードを再配置することが期待される。x=4000 を超えた場合の自動折り返しは本スコープ外とする。
